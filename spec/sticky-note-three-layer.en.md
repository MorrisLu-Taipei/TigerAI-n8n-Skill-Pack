# Sticky Note Three-Layer Structure Spec v1.0

> 🌐 **English** | [繁體中文](sticky-note-three-layer.md)

> **Audience**: end users of this Skill Pack, AI agents, integration engineers
> **Core constraint**: every workflow generated by AI must conform to this three-layer structure

---

## 1. Overview

Each workflow on the n8n canvas is composed of **three vertical lanes**, separated by `position.y`:

```text
┌────────────────────────────────────────────────────────────────┐
│  Layer 1 — USER INTENT (top)              y ≈   -300 ~  -50    │
│  Sticky notes hand-written by the user    color: yellow (4)    │
│  AI treats as "requirement spec"; never modifies                │
├────────────────────────────────────────────────────────────────┤
│  Layer 2 — WORKFLOW NODES (middle)        y ≈     0  ~  500    │
│  AI-generated nodes and connections                             │
│  All Trigger / Logic / Action / Output nodes here              │
├────────────────────────────────────────────────────────────────┤
│  Layer 3 — AI COMMENTARY (bottom)         y ≈   600  ~  900    │
│  Sticky notes AI writes back               color: blue (5)     │
│  Records: node choice rationale, credential needs, test         │
│  recipe, known limits                                           │
└────────────────────────────────────────────────────────────────┘
```

### Why three layers?
- **Reviewable**: a human reviewer instantly distinguishes "intent" / "implementation" / "AI notes".
- **Traceable**: when something goes wrong, compare "user intent" against "AI interpretation" to find the gap.
- **Iterable**: user only updates Layer 1; re-run AI to regenerate Layer 2/3. Layer 1 is always the source of truth.

## 2. Layer 1 — USER INTENT (user writes)

### 2.1 Technical definition
- Node type: `n8n-nodes-base.stickyNote`
- `parameters.color = 4` (yellow)
- `position.y` between `-300` and `-50`
- **AI sees color=4 → knows this is user input → forbidden to modify or move**

### 2.2 Content rules (DSL — see `sticky-note-dsl.md`)

Every Layer 1 sticky must contain at least one DSL block. Minimal parsable example:

```markdown
## Flow intent
@flow: receive order → validate → notify

@trigger: webhook POST /order
@step: validate amount > 0
@step: write to Google Sheet
@step: send Slack notification to #ops
@on-error: email admin@example.com
```

### 2.3 Multiple Layer 1 stickies — ordering
- Multiple Layer 1 stickies are read in `position.x` order (left to right)
- AI treats them as one continuous requirement document

---

## 3. Layer 2 — WORKFLOW NODES (AI writes)

### 3.1 Rules
- All non-sticky-note nodes belong to this layer
- `position.y` between `0` and `500`
- Each node's `notes` field is **required**: one-line mapping to Layer 1's `@step` or `@trigger`
- `connections` must be complete with no orphan nodes

### 3.2 Mapping rules
| Layer 1 DSL | Layer 2 inference |
|---|---|
| `@trigger: webhook ...` | `n8n-nodes-base.webhook` |
| `@trigger: schedule ...` | `n8n-nodes-base.scheduleTrigger` |
| `@trigger: form ...` | `n8n-nodes-base.formTrigger` |
| `@step: validate X` | `n8n-nodes-base.if` or `code` |
| `@step: write to Google Sheet` | `n8n-nodes-base.googleSheets` |
| `@step: send Slack ...` | `n8n-nodes-base.slack` |
| `@step: call API ...` | `n8n-nodes-base.httpRequest` |
| `@step: AI classify ...` | `@n8n/n8n-nodes-langchain.agent` or OpenAI node |
| `@on-error: ...` | Error Trigger workflow or node-level `continueOnFail` |

> Full mapping in `sticky-note-dsl.en.md` §4.

## 4. Layer 3 — AI COMMENTARY (AI writes)

### 4.1 Technical definition
- Node type: `n8n-nodes-base.stickyNote`
- `parameters.color = 5` (blue)
- `position.y` between `600` and `900`
- Each workflow needs **at least one**; recommend multiple stickies aligned with Layer 2 sections

### 4.2 Required content (Markdown template)

```markdown
### 🤖 AI implementation notes — <matching Layer 1 block name>

**Node choices**
- `<node-type>`: rationale (why not alternatives)

**Required credentials**
- [ ] <credential-name>: <which node uses it>

**Assumptions**
- Assumption 1: <e.g., Google Sheet columns already exist>
- Assumption 2: <e.g., Slack channel #ops exists and bot is invited>

**Test recipe**
1. <minimal viable test>
2. <error path test>

**Known limits**
- <e.g., No retry implemented; if API fails, will fail directly>

**Mapped to user requirements**
- @trigger: ... / @step: ...
```

### 4.3 Why mandatory?
- Users understand what decisions AI made without reading code
- Future maintainers (not necessarily AI) get a handover doc
- When something breaks, distinguish "AI misread the requirement" vs "the requirement itself was flawed"

## 5. Color semantics summary

| color value | Use | Who writes |
|---|---|---|
| 4 (yellow) | Layer 1 — user requirement | User |
| 5 (blue) | Layer 3 — AI commentary | AI |
| 7 (green) | (reserved) human ops post-edit | Human ops |
| 6 (red) | (reserved) known BUG / TODO | Anyone |
| 1–3 | not used (avoid ambiguity) | — |

## 6. `@on-error` — two implementation modes

n8n has two error-handling mechanisms. AI must pick one based on Layer 1 description:

### Mode A: node-level `continueOnFail` (within same workflow)
- **For**: single-node failure shouldn't block; downstream branch handles it (e.g. Test 3 Telegram bot)
- **How**: on the failing node set `"continueOnFail": true`; downstream `if` checks `$json.error`
- **Limit**: only catches that node's failure, not whole-workflow exceptions

### Mode B: Error Trigger workflow (cross-workflow)
- **For**: any node failure should alert (e.g. Test 2 scheduled task)
- **How**: separate workflow with `n8n-nodes-base.errorTrigger`; bind to main workflow via "Settings → Error Workflow"
- **Limit**: it's a separate workflow; can't be in same JSON; AI must note "build separate Error Trigger workflow" in Layer 3

### AI selection rules
| Layer 1 phrasing | Use mode |
|---|---|
| "On failure, fallback to action X" / "default value" | A |
| "On failure, alert oncall" / "If whole flow fails, alarm" | B |
| "On failure, retry N times" | node retry config (combined with A) |
| Vague ("@on-error: slack ...") | Try A; if no failing node, switch to B with Layer 3 note |

---

## 7. Violation handling

When AI parses a workflow, in these cases **must refuse and report**:

1. No color=4 sticky note found → "no requirement input"
2. Layer 1 DSL parse fails (no `@trigger`) → "requirement format invalid"
3. color=5 sticky already exists in same workflow → "workflow already AI-processed; overwrite?" (need user confirm)
